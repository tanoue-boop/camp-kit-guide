# E-E-A-T for Affiliate Content

`affiliate-seo` スキルの一部。E-E-A-T（Experience / Expertise / Authoritativeness / Trustworthiness）を高める施策を考える際に参照する。

---

## なぜアフィリ記事に E-E-A-T が重要か

- 商品購入は **金銭的判断**を伴うため YMYL（Your Money or Your Life）に近い領域
- Google は YMYL 領域で E-E-A-T を厳しく評価
- 検索結果上位に残るには **「誰が書いているか」「実体験はあるか」「信頼できるか」** が問われる
- AI 生成コンテンツが増える中、E-E-A-T シグナルが他サイトとの差別化要因になる

---

## Experience（経験）

**実体験から書かれていることを示す。**

### コンテンツでの実装

- 商品の **実使用シーン** を具体的に描写（時期・場所・条件）
- 失敗談や注意点を含める（PR 風でない自然な体験談）
- **体験画像**：実際に使った商品の写真（ストック画像ではなく）
- 「半年使ったレビュー」「3 シーズン経過後の状態」など期間情報

### 商品紹介ブロックでの実装

- 「実際に使ってみて感じたメリット」「気になった点」セクション
- ストック画像のみで構成しない

### 注意

- 体験していないものを「使ってみました」と書かない（虚偽は信頼を損なう）
- 体験がない場合は **「メーカー公式情報・他レビュー・スペックから判断するに」** と明示する

## Expertise（専門性）

**ジャンルに対する深い知識を示す。**

### コンテンツでの実装

- 業界用語の正確な使用（誤用は専門性を損なう）
- 商材ドメインに固有の **判断軸** を明示
  - 例：ナイフなら「刃の素材（ステンレス vs カーボン）」「ロック機構」
  - 例：ランタンなら「ルーメン数の目安」「電源の種類」
  - 例：ホテルなら「立地の利便性」「アメニティ充実度」
- 一般論で終わらせず、商材特有の **「ここを見るべき」** を提示

### 著者情報

- 著者プロフィールを記事ページで明示（または Organization で代替）
- 経歴・資格・実績がある場合は明示
- Person スキーマで構造化データに反映

```tsx
const personSchema = {
  "@context": "https://schema.org",
  "@type": "Person",
  name: "{著者名}",
  description: "{経歴・専門分野}",
  url: "{プロフィールページURL}",
};
```

## Authoritativeness（権威性）

**サイトおよび著者が、そのジャンルで信頼される存在であることを示す。**

### コンテンツでの実装

- **出典明示**：メーカー公式サイト、業界統計、専門メディアの引用
- 引用は外部リンク（`rel="noopener"`）で原典を辿れるように
- 数値や仕様は **メーカー公式情報** に基づく
- 推測や個人意見と、事実情報を **明確に区別**

### サイトレベルの実装

- About ページ・運営者情報ページの整備
- 運営会社の所在地・連絡先
- プライバシーポリシー・利用規約・問い合わせフォーム
- 著者一覧ページ（複数著者の場合）

---

## Trustworthiness（信頼性）

**情報が正確で、最新で、誠実であることを示す。**

### 1. 実データの使用

- 価格・レビュー数・評価は **実データ**（楽天 API・Amazon API 等で取得）
- 取得不可の場合は AggregateRating を **付与しない**（架空データの構造化データ化は禁止）

### 2. 更新日の明示

- 各記事の **更新日** を表示（公開日と更新日を区別）
- BlogPosting スキーマの `dateModified` も同期更新
- 古い情報のまま放置しない

### 3. PR 表記・広告表記

景品表示法・ステマ規制対応：

- アフィリエイト記事には **「PR」「広告」「アフィリエイトリンクを含みます」** 等を明示
- 記事冒頭または商品紹介セクション直前が望ましい
- 表記例：「※本記事にはアフィリエイトリンクが含まれます」

### 4. 透明性

- 商品選定基準を明示
- 「なぜこの 5 製品を選んだか」を冒頭で説明
- 「広告主との関係でランキングを操作していない」旨を運営者情報で明示できれば理想

### 5. 連絡可能性

- 問い合わせフォーム または メールアドレスを公開
- 訂正依頼を受け付ける窓口の整備

## 実装チェックリスト

E-E-A-T を高める施策の確認項目：

### 個別記事レベル

- [ ] 実体験ベースの記述がある（または「公式情報を元に」と明示）
- [ ] 専門用語が正確に使われている
- [ ] 商材ドメイン固有の判断軸が複数提示されている
- [ ] 価格・スペック・レビュー数が実データ（または合理的な参考値）
- [ ] 出典・引用元が明示されている（必要な箇所で）
- [ ] PR / 広告表記がある
- [ ] 公開日と更新日が明示されている
- [ ] アフィリリンクに `rel="nofollow sponsored"`

### サイトレベル

- [ ] About / 運営者情報ページが整備されている
- [ ] プライバシーポリシー・利用規約・問い合わせフォームがある
- [ ] 運営会社・連絡先・所在地が公開されている
- [ ] 著者プロフィールページがある（または Organization で代替）
- [ ] 主要 SNS アカウントが Organization スキーマの `sameAs` に登録されている

## AI 生成コンテンツとの向き合い方

このプロジェクトは AI（Claude）で記事生成を行うため、以下に特に注意：

### AI が「知っているふり」をしないよう制御

- **商品データ**：AI の学習知識ではなく、楽天 API 等の実データを使う
- **価格・レビュー数**：実データで補完。記事執筆時点の参考値である旨を明示
- **「使ってみた」表現**：実体験がない場合は使わない。「公式情報によると」「スペックを見ると」等の客観的表現に置き換え

### AI 生成と分かっても問題ないようにする

Google は **AI 生成自体を NG とはしていない**。ただし、AI 生成でも以下が満たされている必要がある：

- 読者にとって有用
- 正確
- オリジナルな視点・実体験が反映されている
- E-E-A-T シグナルが明示されている

つまり「AI が下書きを作り、人間が体験・専門性・実データで補強する」ハイブリッドが理想形。

## サイト固有設定

各サイトの `CLAUDE.md` で以下を定義：

- 著者を Organization 名義にするか Person にするか
- 記事の更新運用ルール（年 1 回見直し、四半期見直し等）
- PR 表記の文言・配置位置
- 実データ取得元（楽天 API、Amazon API、手動更新等）
- 体験ベース記述が困難なジャンルでの代替策

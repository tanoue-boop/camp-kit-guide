# スキル導入引継ぎ資料

> 対象読者: 既存チャンネルを引き継ぐ Claude、および田之上さん本人  
> 作成日: 2026-04-28  
> 目的: affiliate-seo / affiliate-uiux スキルの既存ワークフローへの安全な導入

---

## 1. 大原則（最重要）

**スキルは既存ワークフローを一切変更しない。**

以下を Claude への命令として宣言する。

- 既存の記事作成手順（KW選定 → GAS → MDX → push の5ステップ）は変更しない
- スキルは品質チェックの観点を提供するだけであり、手順を上書きする権限を持たない
- 田之上さんのいつものアウトプット形式（テーブル・差分サマリー・確認待ち停止）も変更しない
- スキルに書いてある内容を根拠に、既存手順の変更・省略・並べ替えを提案してはならない
- 「スキルに従うから手順を変える」は禁止

スキルが提供するのは「何を確認するか」であり、「どう進めるか」は常に確立済みのワークフローが正である。

---

## 2. スキル導入後の動作

### Claude の内部で起きること

| タイミング | affiliate-seo の参照内容 | affiliate-uiux の参照内容 |
|---|---|---|
| KW選定時 | KW整合性ルール・価格倍率・ブランド多様性の確認観点 | — |
| 商品選定時 | カニバリ回避・想定購入者像との一致確認 | ProductCard の説明文の質確認 |
| MDX生成時 | description の構成・禁止表現チェック | 見出し構造・FAQ の検索意図整合 |
| 自己レビュー時 | スペック表現の断定度・耐水圧不明商品の注記 | 比較表の列選択・読者への情報密度 |

### 田之上さんから見える変化

**増えるもの：** 自己レビューで拾える問題の種類が増える。  
**減るもの：** なし。  
**変わるもの：** なし。

### 田之上さんから見えない変化

裏側で Claude が確認する観点が増えるだけ。指示の出し方も、確認ポイントの置き方も、push までの流れも従来と同一。

## 3. 既存チャンネルへのスキル導入手順

以下の2つを設定するだけで完了する。

1. **スキルの登録**  
   Claude.ai プロジェクト設定から `affiliate-seo` と `affiliate-uiux` スキルを追加する。

2. **CLAUDE.md の配置**  
   リポジトリ直下の `CLAUDE.md` は既に最新版が入っている。Claude Code がリポジトリを開いた時点で自動的に参照される。

これだけ。追加の設定・初期化手順・チャンネルへの説明は不要。

---

## 4. 既存チャンネルでの最初の使い方

**田之上さんは今まで通り指示を出すだけでよい。**

```
例：「いつもの手順で5記事作って。まず KW 候補を出して」
```

スキルが裏で動くため、自己レビューがやや充実するだけ。田之上さん側から意識することは何もない。

もし出力品質に違和感があった時だけ、以下のように追加指示できる。

```
「スキルの観点でもう一度確認して」
```

これはあくまで追加指示であり、最初から全指示をスキル経由にする必要はない。

## 5. このスキル設計チャンネルで起きた事故と回避策

### 事故1：GAS の操作手順の捏造

**何が起きたか：**  
スキル設計チャンネルの初期段階で、Claude が `BATCH_ARTICLES` をスプレッドシートのシート名と誤解し、存在しない操作手順を「それらしく」説明した。

**既存チャンネルで起きるか：**  
起きない。既存チャンネルにはすでに確立した GAS 操作手順があり、Claude が手順を一から考える機会がない。手順が曖昧な場面では「いつもの通りで」という指示で対応できる。

**新規チャンネルを作る時の注意：**  
新しい Claude チャンネルに GAS 操作を依頼する場合は、この引継ぎ資料と `docs/operation-snapshot.md` を最初に共有すること。GAS コードの実体を渡さずに操作手順だけ聞くと同じ事故が起きる。

---

### 事故2：事前選定と楽天実取得結果の大幅乖離

**何が起きたか：**  
スキル運用テスト中に、Claude が事前に想像で固めた5商品（DOD・Coleman・Naturehike 等）と、実際の楽天 Top10 が大きく乖離した。MDX の全面書き直しが必要になった。

**既存チャンネルで起きるか：**  
起きない。既存チャンネルは「GAS で楽天から実取得 → 採用商品を決める」の順序が確立されており、想像商品で MDX を先に書き始めることがない。

**スキル運用テスト時のみ話題になった理由：**  
テスト記事では「GAS 実行前に骨格を書く」という並行作業を試みたため。通常フローでは発生しない。

## 6. 既存チャンネルで「やってはいけないこと」

Claude に対する禁止事項として明記する。

- スキルを引き合いに出して既存手順の変更を提案する
- スキルに記載がないことを理由にアウトプット形式を変える
- 田之上さんに「スキルに従って手順を見直しましょう」と提案する
- 既存の確認停止ポイントをスキル名目で省略する
- スキル導入前と後でチャンネルの進め方が変わったと田之上さんに感じさせる

---

## 7. 既存チャンネルで「やってよいこと」

- 普段通り「いつもの手順で5記事作って」と指示する
- GAS 実行・TSV 配置・MDX 書き換え・ビルド・push の各確認ポイントを置く
- KW整合性チェック（価格倍率5倍以内・ブランド多様性）を自発的に実施する
- 商品紹介本文と「選び方のポイント」の論理整合を自己レビューで確認する
- スペック値が不明な場合は「メーカー公称値参照」「要確認」と注記する
- 出力品質に違和感があった時だけ「スキルの観点で再確認して」と追加指示する

## 8. このスキル設計チャンネルの位置づけ

| 役割 | 担当 |
|---|---|
| スキル・CLAUDE.md・ワークフロードキュメントの整備 | このスキル設計チャンネル |
| 通常の記事作成（KW選定〜push） | 既存チャンネル |
| 改善点の発見・スキル更新 | このスキル設計チャンネル |
| 更新後スキルの利用 | 既存チャンネル（参照するだけ） |

このスキル設計チャンネルは既存チャンネルの記事作成に介入しない。

改善点が見つかった場合の流れ：  
スキル設計チャンネルでスキルを更新 → 既存チャンネルは次回から更新後スキルを参照 → 田之上さんの指示の出し方は変わらない。

---

## 9. 緊急時の戻し方

スキル導入後に問題が起きた場合、以下の手順でいつでもスキル導入前の状態に戻せる。

| 問題 | 戻し方 |
|---|---|
| スキルを外したい | Claude.ai プロジェクト設定からスキルを削除するだけ |
| CLAUDE.md を元に戻したい | `git revert` で対応可能（コミット履歴あり） |
| docs/ 配下のドキュメントを元に戻したい | 同上 |
| チャンネルごと作り直したい | 新チャンネルにこの資料を渡すだけで再現可能 |

スキル導入は不可逆な変更を伴わない。いつでも撤退できる。
